§ 01

Networking Basics

Foundation

A computer network is a collection of interconnected devices (computers, servers, phones, printers, IoT) that exchange data and share resources over wired or wireless links, governed by communication protocols.

Building Blocks

Nodesdevice
Endpoints that send or receive data — computers, servers, mobiles, printers, IoT sensors.
Linksmedium
The path data travels: wired (Ethernet, fiber, coax) or wireless (Wi-Fi, radio, satellite).
Protocolsrules
Agreed rules for communication — TCP/IP, HTTP, FTP, SMTP. Without them, no two devices can talk.

Network Devices — Cheat Sheet

DeviceLayerFunction
HubL1Broadcasts incoming data to all ports. Dumb. Mostly extinct.
SwitchL2Forwards frames to specific MAC addresses within a LAN.
RouterL3Connects different networks; routes packets by IP address.
BridgeL2Connects two LAN segments; filters traffic between them.
GatewaymultiTranslates between different protocols / network architectures.
ModemL1Modulates/demodulates digital data for transmission lines.
Access PointL2Bridges wireless devices into a wired LAN.
FirewallL3–L7Monitors and controls traffic by rule sets; the perimeter cop.

Types of Networks (by Scale)

PAN
Personal Area Network — Bluetooth headphones, smartwatch tethering. Range: ~10m.
LAN
Local Area Network — Single office, building, or home. Privately owned. High-speed, low-latency.
CAN
Campus Area Network — Multiple LANs across a campus (e.g., IIFT Kakinada).
MAN
Metropolitan Area Network — Spans a city or town.
WAN
Wide Area Network — Countries / continents. The Internet itself is the largest WAN.
VPN
Virtual Private Network — Encrypted tunnel over a public network.
SAN
Storage Area Network — High-speed access to centralized storage.

Topology — How Devices Connect

Buslinear
All devices on one shared cable. Cheap, but a single cable break kills the network.
Starcentralized
All devices connect to a central hub/switch. Most common in modern LANs.
Ringcircular
Each device connects to two others, forming a loop. Token Ring legacy.
Meshresilient
Every node connects to many others. Fault-tolerant but expensive. Links = N(N−1)/2.
Treehierarchical
Hybrid of bus + star. Root → branches. If root fails, network fails.
Hybridmixed
Combines two or more topologies. What real-world enterprise networks actually look like.

Architecture Models

Client–Server
Centralized server provides resources; clients request them. e.g., the web.
Peer-to-Peer (P2P)
All devices equal — each acts as both client and server. e.g., BitTorrent.

Switching Techniques

  • Circuit switching — Dedicated path reserved for the entire session. Predictable but wastes idle capacity. (e.g., traditional telephone PSTN)
  • Packet switching — Data broken into packets; each routed independently. Efficient, what the Internet uses.
  • Message switching — Whole message stored and forwarded node-to-node. Legacy.

Key Network Criteria

  • Performance — Measured by transit time, throughput, response time.
  • Reliability — How often the network fails and how fast it recovers.
  • Security — Protection from unauthorized access, viruses, eavesdropping.
Real-world analogy The postal system as a network

Your house = node. Roads = links. Postal rules (envelope format, stamps, addresses) = protocols. The post office sorting facility = router. Your neighborhood mailman = switch (local delivery). A courier handing off to another courier in a different city = gateway. The same letter could be routed through several cities (packet switching) — and you don't care about the path as long as it arrives.

Exam tip The 5 things examiners love asking
  1. Difference between hub, switch, and router (always tested).
  2. LAN vs MAN vs WAN — give one example each.
  3. Mesh topology link formula: N(N-1)/2.
  4. Client-server vs P2P — at least two differences.
  5. Why packet switching beat circuit switching for the internet.
§ 02

TCP 3-Way Handshake

Transport Layer

TCP (Transmission Control Protocol) is a connection-oriented protocol at the transport layer. Before any data flows, both ends synchronize sequence numbers and confirm they can hear each other through a structured three-message exchange known as the 3-way handshake — SYN → SYN-ACK → ACK.

The Three Steps

STEP 1 SYN — “Hey, can we talk?”
Client sends a TCP segment with the SYN flag set and an Initial Sequence Number (ISN, e.g. seq=x). It enters the SYN-SENT state. The SYN packet carries no application data but consumes one sequence number.
STEP 2 SYN-ACK — “Yes, here's my number too.”
Server responds with both SYN and ACK flags set. ack=x+1 confirms the client's SYN. Server picks its own ISN (seq=y). Server transitions from LISTEN to SYN-RECEIVED.
STEP 3 ACK — “Got it, let's go.”
Client replies with ACK, seq=x+1, ack=y+1. Both sides move to ESTABLISHED. Full-duplex communication begins; data segments now flow in both directions.

What's Actually Negotiated

  • Initial Sequence Numbers (ISNs) for both directions — randomized for security (predictable ISNs = TCP hijacking risk).
  • Window size — how much data each side can buffer (flow control).
  • Maximum Segment Size (MSS) — largest TCP payload per packet, typically 1460 bytes.
  • Optional features — selective acknowledgement (SACK), window scaling, timestamps.

TCP State Machine — Connection Setup

CLOSED → SYN-SENT → ESTABLISHED · LISTEN → SYN-RECEIVED → ESTABLISHED

Visualize It with a Real Packet Capture

wireshark · tcp handshake
$ tshark -i any -f 'tcp port 80' -w handshake.pcap # Frame 1 (Client → Server) 192.168.1.10 → 10.0.0.1 TCP 74 [SYN] seq=0 win=64240 MSS=1460 # Frame 2 (Server → Client) 10.0.0.1 → 192.168.1.10 TCP 74 [SYN, ACK] seq=0 ack=1 win=65535 MSS=1460 # Frame 3 (Client → Server) 192.168.1.10 → 10.0.0.1 TCP 66 [ACK] seq=1 ack=1 win=64240 # ESTABLISHED — HTTP traffic now flows.

4-Way Handshake — Connection Termination

Closing is asymmetric because each side closes its half independently:

  1. Side A sends FIN → enters FIN-WAIT-1.
  2. Side B sends ACK → A enters FIN-WAIT-2, B enters CLOSE-WAIT.
  3. Side B sends its own FIN → enters LAST-ACK.
  4. Side A sends final ACK → enters TIME-WAIT, then CLOSED.

Why Reliable? — Positive Acknowledgement with Retransmission (PAR)

Every segment sent expects an ACK. If lost, corrupted, or unacknowledged within the timeout, the sender retransmits. The checksum field catches in-flight corruption; the receiver silently drops bad segments and waits for the resend.

attack vector

SYN Flood (DoS) — Attacker sends many SYN packets but never completes the handshake (never sends the final ACK). Server's half-open connection table fills up, exhausting resources. Mitigation: SYN cookies, rate limiting, firewall stateful inspection.

Real-world analogy The phone call

You call your friend (SYN). They pick up and say "Hello?" (SYN-ACK). You say "Hey, it's me." (ACK). Now you can actually talk. If you skip step 3 and just start talking, your friend has no idea you actually heard them.

Exam tip 5-mark answer template

(1) Define TCP as connection-oriented. (2) List the 3 steps with flags and seq/ack numbers. (3) Mention ESTABLISHED state. (4) Draw a simple diagram (always draws marks). (5) Bonus: mention SYN flood as a known attack on the handshake.

§ 03

OSI Model — Open Systems Interconnection

Reference Model

A conceptual framework developed by the ISO in 1984 that splits network communication into seven abstraction layers. Each layer has a defined responsibility, talks only to the layer above and below it, and abstracts away the layer below. While the real-world internet runs on TCP/IP, OSI is the universal language for discussing networking and security.

The Seven Layers (Top → Bottom)

L7
Application
User-facing protocols. Where browsers, email clients, and apps interact with the network.
HTTP · HTTPS · FTP · SMTP · DNS
L6
Presentation
Translation, encryption/decryption, compression. Makes data readable for the app layer.
SSL/TLS · JPEG · MPEG · ASCII
L5
Session
Establishes, manages, terminates communication sessions between applications.
NetBIOS · RPC · PPTP
L4
Transport
End-to-end delivery, segmentation, flow control, error recovery. TCP = reliable, UDP = fast.
TCP · UDP
L3
Network
Logical addressing & routing. Decides the path packets take across networks.
IP · ICMP · IPsec
L2
Data Link
Node-to-node delivery on the same network. Framing, MAC addressing, error detection.
Ethernet · ARP · PPP
L1
Physical
Raw bits over the wire/airwaves. Cables, voltages, signal timing, hardware specs.
Cables · Wi-Fi · Fiber · Hubs

Memorization Mnemonics

Top → BottomL7 → L1
All People Seem To Need Data Processing
Bottom → TopL1 → L7
Please Do Not Throw Sausage Pizza Away

Data Encapsulation — What Each Layer Adds

L7 Data → L4 Segment → L3 Packet → L2 Frame → L1 Bits

Data travels down the stack on the sender, gets wrapped in headers at each layer, traverses the medium as bits, then climbs back up the stack on the receiver — unwrapping headers along the way.

Why OSI Still Matters for Security

Threats and defenses map to layers:

  • L2 — ARP spoofing, MAC flooding → port security, dynamic ARP inspection.
  • L3 — IP spoofing, packet filtering attacks → firewalls, ACLs.
  • L4 — SYN floods, port scanning → stateful inspection, rate limits.
  • L7 — SQL injection, XSS, malware → WAFs, app-aware filtering, input validation.
Real-world analogy Sending a letter

L7: you decide to write a letter. L6: you write it in English (encoding). L5: you agree with your friend on letter-writing cadence (session). L4: you split a long letter into pages (segments). L3: you write the destination address (routing). L2: you put it in an envelope and seal it (framing). L1: the postal truck carries it (physical transmission). The recipient reverses every step.

Exam tip The "give one protocol per layer" question

Examiners always ask "Name one protocol from each layer." Lock in: HTTP (L7), SSL/TLS (L6), NetBIOS (L5), TCP (L4), IP (L3), Ethernet (L2), cables/Wi-Fi (L1). Also remember the PDU at each layer: data → segment → packet → frame → bits.

§ 04

How the Internet Works

Foundation

The Internet is a global network of networks — millions of interconnected LANs, WANs, data centers, routers, satellites, and submarine cables, all speaking TCP/IP. No single owner. No central control. Just an agreement on protocols.

The Lifecycle of a Web Request — What Happens When You Type "google.com"

1DNS Lookup
Browser asks a DNS resolver: "What's the IP for google.com?" It returns something like 142.250.182.78. DNS is the phonebook of the internet.
2TCP Handshake
Browser opens a TCP connection to the server's IP on port 443 (HTTPS). 3-way handshake completes.
3TLS Handshake
Browser and server negotiate encryption, exchange certificates, and derive session keys. From here, all traffic is encrypted.
4HTTP Request
Browser sends GET / HTTP/1.1 with headers (User-Agent, cookies, etc.). Server receives it.
5Server Processes & Responds
Server fetches the page (often from cache), generates HTML, sends back HTTP/1.1 200 OK + body.
6Browser Renders
Parses HTML, fetches CSS/JS/images (more requests!), paints the page. You see google.com in ~300ms.

Core Components of the Internet

ISPaccess
Internet Service Provider. Connects your home/office to the global internet. e.g., Jio, Airtel, BSNL.
DNSnaming
Domain Name System. Translates domain names → IP addresses. Distributed, hierarchical.
Routersforwarding
Forward packets between networks using IP. The internet's traffic cops.
Backboneinfra
High-capacity fiber links + submarine cables connecting continents. The internet's spine.
IXPspeering
Internet Exchange Points — physical locations where ISPs interconnect to swap traffic.
Data Centershosting
Where servers live. Google, AWS, Cloudflare run massive ones globally.

IP Addresses — The Internet's Postal Code

IPv4
32-bit address, e.g. 192.168.1.1. ~4.3 billion possible — already exhausted.
IPv6
128-bit, e.g. 2001:0db8::1. 340 undecillion addresses. The future.
Public IP
Routable on the internet. Assigned by ISP.
Private IP
Used inside LANs only (10.x, 172.16-31.x, 192.168.x). Not internet-routable.
NAT
Network Address Translation. Your router maps private IPs to one public IP — how multiple devices share one ISP connection.

Internet ≠ World Wide Web

Common confusion: the Internet is the infrastructure — the network of networks. The Web (WWW) is just one service running on it (websites, accessed via HTTP/HTTPS in browsers). Email, video calls, gaming, DNS, file sharing — all separate services on the same internet.

Cybersecurity 101

Overview

Cybersecurity is the practice of protecting systems, networks, programs, and data from digital attacks aimed at accessing, changing, or destroying sensitive information; extorting money from users; or interrupting normal business processes.

Why Cybersecurity Matters — The Stakes

  • Average cost of a data breach (2024): USD 4.88 million (IBM).
  • Global cybercrime damages projected to hit USD 10.5 trillion by 2025 (Cybersecurity Ventures).
  • India: 1.5 million+ cybercrime complaints filed in 2023; financial fraud alone caused ₹7,000+ crore losses.

Domains of Cybersecurity

Protecting laptops, mobiles, servers — antivirus, EDR, device management.
Operational Securitypeople
Process & decisions for handling/protecting data assets. Includes user training.
Identity & Accessauthn

Disaster Recovery / BCPresilience
How an org recovers from cyber incidents and natural disasters. Covered in §18.

Threat Actors

  • Hacktivists — Politically/socially motivated (e.g., Anonymous).
  • Cybercriminals — Profit-driven. Ransomware operators, fraud rings.

§ 06

The CIA Triad

Core Model

The CIA Triad is the foundational model of information security. Three pillars — Confidentiality, Integrity, Availability — that every security control, policy, or framework ultimately maps to. If even one is missing, the security posture is incomplete.

C
Confidentiality
Only authorized people, processes, or systems can access information. Prevents unauthorized disclosure.
controls: encryption · access control
mfa · classification · masking
I
Integrity
Data remains accurate, complete, and unaltered — except by authorized parties. Prevents unauthorized modification.
controls: hashing · digital signatures
checksums · version control · audit logs
A
Availability
Authorized users can access systems and data when needed. Prevents denial of service / outages.
controls: redundancy · backups
ddos mitigation · uptime monitoring

Real Examples for Each

Confidentiality
Multi-factor authentication on banking login. Encrypted WhatsApp messages. Salary data locked to HR.
Integrity
Software updates signed with publisher's private key. Hash check after downloading a file. Bank balance can't be changed by anyone except authorized transactions.
Availability
Amazon's 99.99% uptime promise. Hospital server failover during a power cut. Cloudflare absorbing a DDoS so your site stays up.

The Tension Between the Three

The three goals are sometimes in conflict — optimizing for one can hurt another:

  • C vs A: Strict access controls protect confidentiality but can frustrate availability if users get locked out.
  • I vs A: Aggressive integrity checks may pause operations until verified, hurting availability.
  • Use-case priority:
    • Banking → Integrity first (can't have wrong balances)
    • Defense intel → Confidentiality first (secrets must stay secret)
    • E-commerce on Black Friday → Availability first (downtime = millions lost)

Threats Mapped to the Triad

ThreatPillar BrokenWhy
Data breach / leakConfidentialitySensitive data disclosed to unauthorized parties.
Eavesdropping / sniffingConfidentialityTraffic intercepted without consent.
Ransomware (encryption)AvailabilityLegitimate users locked out of their own files.
DDoS attackAvailabilityService overwhelmed, can't respond to real users.
Data tamperingIntegrityRecords modified without authorization.
MitM with injectionIntegrity + ConfidentialityAttacker reads & modifies traffic in flight.

Extended Models (Beyond CIA)

Some practitioners add more pillars for completeness:

  • Authenticity — Verifying users/data are who/what they claim.
  • Non-repudiation — A user can't deny having performed an action (digital signatures).
  • Accountability — Actions can be traced back to identities (audit logs).

Together with CIA, these form the "Parkerian Hexad" (Donn Parker, 1998).

Exam tip How to score on a CIA question

(1) Define each pillar in one line. (2) Give one example for each. (3) Give one control/mechanism for each (encryption, hashing, redundancy). (4) Mention the trade-offs. (5) If 10-mark, add Parkerian Hexad. Always draw the triangle diagram — easy marks.

§ 07

Cryptography & Its Types

Core Defense

Cryptography is the practice of securing information by transforming it into an unreadable form so only authorized parties can read it. The original message is plaintext; the scrambled output is ciphertext. The transformation uses an algorithm + key.

The Three Types

Symmetric Key1 key
Same key encrypts & decrypts. Fast, efficient. Problem: secure key exchange. e.g., AES, DES, 3DES.
Asymmetric Key2 keys
Public key encrypts, private key decrypts. Solves key-exchange problem. Slower. e.g., RSA, ECC, Diffie-Hellman.
Hash Functionsno key
One-way transform → fixed-length digest. Not for confidentiality — for integrity & verification. e.g., SHA-256, MD5 (broken).

Symmetric Encryption — Deep Dive

The same secret key is used at both ends. Both sender and receiver must agree on the key beforehand and keep it hidden from everyone else.

Plaintext → Encrypt (key K) → Ciphertext → Decrypt (key K) → Plaintext
  • Strengths: Fast, low computational cost, great for bulk encryption (full-disk encryption, large file transfers).
  • Weakness: Key distribution. If the key leaks, all messages are compromised. With N users wanting to communicate, you need N(N-1)/2 keys total.
  • Algorithms: AES (128/192/256-bit — current standard), DES (56-bit, broken), 3DES (legacy), Blowfish, RC4 (deprecated).

Asymmetric Encryption — Deep Dive

Each user has a key pair: a public key (shared with the world) and a private key (kept secret). What one encrypts, only the other can decrypt.

Plaintext → Encrypt (Bob's public key) → Ciphertext → Decrypt (Bob's private key) → Plaintext
  • Strengths: No need to securely share keys beforehand. Also enables digital signatures (sign with private, verify with public).
  • Weakness: Slow. Computationally expensive. Vulnerable to future quantum computers.
  • Algorithms: RSA (most common), ECC (elliptic curve — same security, smaller keys), Diffie-Hellman (key exchange), ElGamal, DSA (signatures).

Hashing — Deep Dive

Hashing turns any input into a fixed-length, irreversible digest. The output is unique-ish — even a 1-bit change in input produces a wildly different hash. Hashes are used for integrity verification, password storage, and blockchain.

sha256 demo
$ echo -n 'hello' | sha256sum 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824 $ echo -n 'Hello' | sha256sum # changed one letter 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
  • Properties: deterministic · one-way · fixed output length · avalanche effect · collision-resistant.
  • Uses: file integrity checks · password storage (with salt) · digital signatures · blockchain (Merkle trees).
  • Algorithms: SHA-256, SHA-3, BLAKE2. Avoid: MD5, SHA-1 (collision vulnerabilities).

Real-World Hybrid Crypto — How HTTPS Actually Works

HTTPS combines all three for the best of each:

  1. Asymmetric (RSA/ECC) is used to securely exchange a temporary symmetric session key.
  2. Symmetric (AES) then encrypts the actual data — much faster.
  3. Hashing (SHA-256) verifies integrity via MAC/HMAC.

Symmetric vs Asymmetric — Quick Compare

AspectSymmetricAsymmetric
Keys used1 shared2 (public + private)
SpeedFastSlow
Key distributionHardEasy
Best forBulk data encryptionKey exchange, digital signatures
Key length128–256 bits2048–4096 bits (RSA)

looming threat

Quantum cryptography risk — Future quantum computers could break RSA and ECC by efficiently solving integer factorization and discrete log problems. NIST is finalizing post-quantum cryptography standards now. Symmetric algorithms (AES-256) are mostly safe.

§ 08

Identity & Access Management (IAM)

Access Control

IAM is the framework of policies, processes, and technologies that ensures the right people have the right access to the right resources at the right time — and that everyone else stays out. The goal: enable productivity while denying attackers.

The IAAA Framework

Identification
A user claims an identity — typically via a username. "I am Harsh."
Authentication (AuthN)
Proving the claim is true — via password, biometric, token. "Prove it."
Authorization (AuthZ)
Determining what an authenticated user can do — permissions, roles. "What can you access?"
Accountability
Logging actions so they can be traced back — audit trails. "Who did this?"

Authentication Factors

Something you knowType 1
Password, PIN, security question. Cheapest but weakest — guessable, phishable.
Something you haveType 2
OTP device, smartcard, security key (YubiKey), phone for SMS code.
Something you areType 3
Biometrics — fingerprint, face, iris, voice. Hard to replicate but irrevocable.
Somewhere you areType 4
Geolocation / IP-based context (e.g., reject logins from new countries).
Something you doType 5
Behavioral biometrics — typing rhythm, mouse movement patterns.

Multi-Factor Authentication (MFA)

Combines two or more factors from different categories. Even if a password is stolen, the attacker can't log in without the second factor. MFA is one of the single highest-ROI security controls — Microsoft reports it blocks 99.9% of automated account attacks.

2FA (two-factor) is the most common form — password + OTP. Adaptive MFA triggers based on risk (new device, unusual location).

Single Sign-On (SSO)

Lets a user authenticate once and access multiple connected applications without re-entering credentials. Reduces password fatigue, centralizes identity management. Common protocols:

  • SAML — XML-based, common in enterprise.
  • OAuth 2.0 — delegation protocol ("Sign in with Google").
  • OpenID Connect — identity layer on top of OAuth 2.0.
  • Kerberos — ticket-based, used inside Windows Active Directory.

Authorization Models

ModelHow it worksBest for
DAC (Discretionary)Resource owner decides who gets accessSmall teams, personal files
MAC (Mandatory)System enforces access based on classification labels (top secret, etc.)Military, government
RBAC (Role-Based)Access tied to job role (HR, Dev, Admin)Most enterprises
ABAC (Attribute-Based)Access based on user attributes + resource attributes + contextDynamic, complex orgs
PBAC (Policy-Based)Centralized policies evaluated at runtimeCloud-native, zero trust

The Principle of Least Privilege (PoLP)

Give users only the minimum access they need to perform their job — nothing more. Same for processes and services. Reduces blast radius of a compromise. Closely related: JIT (Just-in-Time) access grants permissions temporarily and revokes them after.

Identity Lifecycle Management

1Provisioning
Create account when user joins — assign role, baseline permissions.
2Maintenance
Update permissions as roles change. Access reviews quarterly.
3Deprovisioning
Revoke access immediately when user leaves. Orphaned accounts = breach vectors.

Privileged Access Management (PAM)

A specialized subset of IAM focused on administrator and superuser accounts. PAM solutions provide vaulting, session recording, just-in-time elevation, and approval workflows. Per BeyondTrust research, 75% of critical Microsoft vulnerabilities could be mitigated by removing admin rights — proof that PAM matters.

Key Components of a Modern IAM System

  • Identity Provider (IdP) — central directory holding identities (e.g., Microsoft Entra ID, Okta).
  • Directory Service — stores user attributes (Active Directory, LDAP).
  • SSO & Federation — share identities across domains.
  • MFA enforcement.
  • Access governance — review and certify access periodically.
  • Audit & reporting.
zero trust principle

"Never trust, always verify." Zero Trust assumes attackers are already inside the network. Every request — even from "internal" users — gets continuously authenticated, authorized, and inspected. IAM is the foundation of Zero Trust.

§ 09

Vulnerabilities

Risk Surface

A vulnerability is a weakness — in software, hardware, configuration, or human process — that a threat actor can exploit to compromise the CIA of a system. Three concepts to keep distinct:

Vulnerability
The weakness itself (unpatched software, weak password).
Threat
A potential cause of harm that could exploit the vulnerability (a hacker, malware).
Risk
The intersection — likelihood × impact. Risk = Threat × Vulnerability × Impact.

Common Vulnerability Categories

Unpatched Softwarecritical
Known CVEs in OS, libraries, apps that haven't been updated. The #1 enterprise vulnerability.
Misconfigurationshigh
Default passwords, open ports, exposed buckets, weak TLS. Often free wins for attackers.
Weak Authenticationhigh
Weak passwords, no MFA, reused credentials across services.
Injection Flawscritical
SQLi, command injection, XSS — user input executed as code.
Insecure APIshigh
Endpoints without auth, broken object-level access (BOLA), excessive data exposure.
Zero-Dayscritical
Unknown vulnerabilities — no patch exists yet. Highly valuable on dark markets.
Insider Riskmedium
Disgruntled or careless employees with legitimate access.
Physical Accessmedium
Unattended laptops, USB drops, tailgating into secure areas.

The Vulnerability Management Lifecycle

1Discover
Inventory all assets — servers, endpoints, cloud services, apps. You can't protect what you don't know exists.
2Assess / Scan
Automated scanners (Nessus, Qualys, OpenVAS) plus manual review identify known CVEs and misconfigurations.
3Prioritize
Rank by severity (CVSS), exploitability, asset criticality. Risk-based — fix what attackers will actually use.
4Remediate
Apply patches, change configs, deploy compensating controls. Track via tickets with SLAs.
5Verify
Rescan to confirm the fix worked. Don't trust — verify.
6Report & Improve
Metrics: MTTR (Mean Time to Remediate), coverage, repeat findings. Feed back into the cycle.

9.0 – 10.0CriticalPatch immediately / hours

CVE — The Universal Vulnerability ID

Each known vulnerability gets a unique CVE ID (e.g., CVE-2021-44228 — the famous Log4Shell). Maintained by MITRE; referenced in patches, advisories, scanners. Critical for cross-tool communication.

Next-Gen / Risk-Based Vulnerability Management (RBVM)

Modern programs go beyond patch-everything by combining:

  • Threat intelligence — is this CVE actually being exploited in the wild?
  • Exploitability — does public exploit code exist?
  • Asset criticality — is this on the public-facing payment server or a dev box?
  • Compensating controls — is there a WAF rule that blocks the exploit path?

The result: a prioritized fix list focused on real risk, not just raw count.

attack timeline

Average time from a CVE disclosure to exploitation in the wild is shrinking every year. Critical vulns are now weaponized within hours, not weeks. Speed of patching is a competitive advantage.

§ 10

Security Testing

Validation

Security testing is the systematic evaluation of an application, system, or network to identify vulnerabilities, weaknesses, and gaps before attackers do. It validates that security controls actually work — not just that they exist on paper.

The Goals — What Security Testing Verifies

Each goal maps to a CIA pillar plus three more practical properties:

  • Confidentiality — data is accessible only to authorized users.
  • Integrity — data and code haven't been tampered with.
  • Authentication — users and systems are who they claim to be.
  • Authorization — users can only do what they're permitted.
  • Availability — services stay up under attack/load.
  • Non-repudiation — actions are provably attributable.

Types of Security Testing

Vulnerability Scanningautomated
Tools scan systems for known weaknesses against a database (CVEs). Fast, broad, surface-level. e.g., Nessus, Qualys.
Penetration Testingmanual
Humans simulate real attacks — exploit chains, lateral movement. Deeper than scanning. Covered in §11.
Risk Assessmentanalytical
Identifies threats, estimates impact, prioritizes mitigations. Strategic.
Security Auditcompliance
Checks the system against standards/policies (ISO 27001, PCI-DSS). Code review + control verification.
Ethical Hackingoffensive
Authorized hacking to find security holes before criminals do. Broader than pentest.
Posture Assessmentholistic
Evaluates overall security maturity — policies, tech, people, process. Big-picture view.

Application Security Testing (AST)

SAST
Static Application Security Testing — analyzes source code without running it. White-box. Finds insecure patterns early.
DAST
Dynamic Application Security Testing — tests running app from outside (black-box). Finds runtime issues like auth flaws, injection.
IAST
Interactive — agent inside the app monitors live execution. Combines SAST + DAST.
SCA
Software Composition Analysis — scans third-party libraries/dependencies for known CVEs.
RASP
Runtime Application Self-Protection — agent that detects and blocks attacks in production.

The Testing Process

1Planning
Define scope, goals, rules of engagement. What's in/out of bounds.
2Reconnaissance
Gather information about the target — passive (OSINT) or active (scanning).
3Testing & Exploitation
Run automated tools + manual techniques. Attempt to exploit findings.
4Reporting
Document findings with severity, evidence, reproduction steps, fix recommendations.
5Retest
Verify fixes after remediation. Close the loop.

Bug Bounty Programs

Crowdsourced security testing. Companies invite ethical hackers worldwide to find and report bugs in exchange for cash rewards. Platforms like HackerOne and Bugcrowd coordinate. Complements internal testing with diversity of perspective.

Penetration Testing

Offensive Security

A penetration test ("pen test") is an authorized simulated cyberattack on a system to identify exploitable vulnerabilities and assess the real-world impact of a breach. Where vulnerability scanning answers "what's wrong?", pen testing answers "how badly can someone get in?"

Pen Testing vs Vulnerability Scanning

AspectVuln ScanPen Test

DepthBreadthDepth
ExploitsIdentifies onlyActually exploits to prove impact
FrequencyContinuous / monthlyAnnually or after major changes

The Three Boxes — Pen Test Perspectives

Black Boxexternal
Tester has zero knowledge — simulates an outside attacker. Most realistic. Slowest. Best simulates real cybercriminals.
Gray Boxpartial
Limited knowledge — credentials of a regular user given. Simulates a malicious insider or compromised low-privilege account.
White Boxfull
Tester has full access — source code, architecture diagrams, admin creds. Deepest insight; most efficient.

Types of Pen Tests by Target

  • Network — Internal & external network infrastructure.
  • Web Application — OWASP Top 10, business logic flaws.
  • Mobile — iOS, Android apps and their backends.
  • Wireless — Wi-Fi security, rogue access points.
  • Physical — Tailgating, lock picking, badge cloning.
  • Social Engineering — Phishing simulations, vishing.
  • Cloud — AWS/Azure/GCP misconfigurations, IAM weaknesses.
  • Red Team — Full-scope, multi-month, "anything goes" engagement simulating an APT.

The 5 Phases of a Penetration Test

PHASE 1Planning & Reconnaissance
Define scope, rules of engagement, success criteria. Gather intel on the target — domain info, employee names, tech stack (passive OSINT) and active probing.
PHASE 2Scanning & Discovery
Map the attack surface — port scans (nmap), service detection, vulnerability scans, enumerate users and shares.
PHASE 3Gaining Access (Exploitation)
Exploit identified vulnerabilities — SQL injection, password attacks, exploit code (Metasploit), social engineering. The goal: establish a foothold.
PHASE 4Maintaining Access & Lateral Movement
Escalate privileges, install persistence, move laterally to other systems. Chain vulnerabilities. Simulate APT-like long-term presence.
PHASE 5Analysis & Reporting
Clean up — remove tools, restore configs. Deliver report: executive summary, technical findings with proof, business impact, prioritized recommendations.

§ 12

Social Engineering

Human Layer

Social engineering is the art of psychological manipulation — tricking people into giving up confidential information, granting access, or performing actions that compromise security. It bypasses technical defenses by exploiting the weakest link: humans. Per Proofpoint research, over 90% of cyberattacks start with social engineering.

Why It Works — The Psychology

Attackers exploit cognitive biases and emotional triggers:

  • Authority — "I'm from IT / your CEO." People comply with perceived authority.
  • Urgency — "Pay now or your account closes in 1 hour." Panic kills critical thinking.
  • Fear — "We've detected suspicious activity." Triggers reactive behavior.
  • Greed — "You've won a lottery!" Promise of reward.
  • Curiosity — "Click here to see what your ex posted about you."
  • Helpfulness — "Can you hold the door for me? My hands are full." Social politeness.
  • Trust / familiarity — Spoofing colleagues, brands, friends.
  • Fatigue — Timing attacks for when targets are tired or distracted.

Common Attack Types

Phishingemail
Mass emails impersonating trusted brands to harvest credentials or deliver malware. The most common form.
Spear Phishingtargeted
Personalized phishing — researched to target a specific individual or org. Much higher success rate.
Whalingexec
Spear phishing targeting C-suite executives. High-value, high-impact.
Smishingsms
Phishing via SMS — "Your package is held; pay ₹50 here." Big in India.
Vishingvoice
Voice phishing — fake bank, customs, or "tech support" calls demanding info or payment.
BECcorporate
Business Email Compromise — attacker impersonates an executive, instructs finance to wire funds.
Pretextingscenario
Inventing a fake scenario ("I'm from audit; need your password to verify"). Foundation of many attacks.
Baitingphysical
Leaving infected USB drives in parking lots labeled "Confidential — Salary 2024". Curiosity does the rest.
Tailgatingphysical
Following an authorized person into a secure area without scanning a badge.
Quid Pro Quotrade
Offering a service in exchange for information — "Free tech support, just share your screen."
Angler Phishingsocial
Fake support accounts on Twitter/Insta intercepting customer complaints to steal credentials.
Deepfakeai
AI-generated voice/video impersonating real people. Already used in CEO fraud.

Phishing Indicators — How to Spot One

  • Generic greetings ("Dear Customer") instead of your name.
  • Urgency / threats of account suspension.
  • Suspicious sender address — slight misspellings (amaz0n.com, support@amazom.com).
  • Unexpected attachments, especially .zip, .exe, .html.
  • Links that don't match displayed text (hover before clicking).
  • Requests for credentials, OTPs, payment info via email.
  • Grammar or formatting errors (less reliable with AI now).
  • Pressure to bypass normal procedures ("don't tell anyone, urgent").

Defenses

Awareness Training
Regular phishing simulations + education. The #1 control. Microsoft reports trained users are far less likely to click.
Email Filtering
SPF, DKIM, DMARC for sender verification. Anti-phishing AI gateways (Proofpoint, Mimecast).
MFA
Even if credentials are phished, attacker can't log in without second factor.
Verification Procedures
Always confirm payment/access requests through a separate channel — call the person, don't email back.
Zero Trust
Don't trust internal traffic by default. Limit blast radius of compromised accounts.
india context

Digital arrest scams exploded in 2024–2025 in India — fraudsters impersonate CBI/ED officers via video call, threaten arrest, and pressure victims into transferring money "to clear their name." Police never demand money over video. This is social engineering at its rawest.

§ 13

Digital Forensics & Incident Response (DFIR)

Response

DFIR fuses two disciplines into one workflow:

  • Digital Forensics — Investigating cyber incidents to gather, analyze, and preserve digital evidence — often for legal proceedings.
  • Incident Response (IR) — Detecting, containing, eradicating, and recovering from active cyberattacks.

Run separately, they interfere: IR teams accidentally destroy evidence while removing threats; forensics teams delay containment by gathering data slowly. DFIR unifies them — preserve evidence while stopping the bleeding.

Why DFIR Matters

  • Faster recovery — IBM's 2024 data shows organizations with strong IR teams save an average USD 473,706 per breach.
  • Legal admissibility — Evidence collected with proper chain of custody can stand up in court.
  • Root cause analysis — Identifies how the breach happened so it can't repeat.
  • Compliance — Regulators (GDPR, DPDP, HIPAA) require breach documentation and reporting.

Incident Response Lifecycle (NIST 4-Phase Model)

PHASE 1Preparation
Build the IR plan, train the team, deploy tools (SIEM, EDR), establish communication channels. The proactive groundwork done before an incident.
PHASE 2Detection & Analysis
Monitor for indicators of compromise (IoCs). Validate alerts. Determine scope and severity. Begin documentation.
PHASE 3Containment, Eradication, Recovery
Short-term containment (isolate infected hosts) → eradicate threat (remove malware, close attack vectors) → recover (restore from clean backups, validate).
PHASE 4Post-Incident Activity
Lessons learned. Update IR plan. Improve controls. File regulatory reports. Conduct retrospective.

SANS uses a similar 6-phase model: Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned.

Digital Forensics Process

1Identification
Determine what evidence exists, where it lives, and what's relevant.
2Collection / Preservation
Make bit-for-bit copies (forensic images) of disks, memory, network captures. Hash everything to prove integrity. Never modify the original.
3Examination
Process the data — extract artifacts, recover deleted files, analyze logs.
4Analysis
Correlate evidence to reconstruct the timeline and attack methodology. Answer: who, what, when, where, how.
5Reporting
Compile findings, attribution if possible, recommendations. Format for both technical and legal audiences.

Types of Digital Forensics

Disk Forensicsstorage
Analyzing hard drives, SSDs — file system artifacts, deleted files, slack space.
Memory ForensicsRAM
Analyzing live memory dumps — running processes, encryption keys, credentials, fileless malware.
Network Forensicstraffic
Packet captures, flow records, DNS logs — trace data exfiltration paths, C2 communications.
Mobile Forensicsdevice
Phones & tablets — call logs, messages, app data, location history.
Cloud Forensicscloud
CloudTrail logs, API calls, snapshots — challenging due to ephemeral nature.
Malware Forensicscode
Reverse engineering malware to understand behavior, IoCs, attribution.

Chain of Custody — The Legal Backbone

A documented trail proving who handled the evidence, when, where, and why — from collection to court. If broken, evidence becomes inadmissible. Every transfer signed, dated, timestamped. Original media never modified — only forensic copies are analyzed.

Cyber Warfare

Nation-State

Cyber warfare is a cyberattack — or series of attacks — that targets a country, with the potential to wreak havoc on government and civilian infrastructure and disrupt critical systems. Damage can range from data loss to loss of life. There is no universal formal definition; even the US DoD recognizes the threat but stops short of a precise scope.

Who's Behind It

  • Nation-states vs nation-states — the classical model (Russia vs Ukraine, US vs Iran, China vs everyone).
  • Non-state actors — terrorist organizations, hacktivist collectives.
  • Proxy actors — criminal groups operating with implicit state backing.

§ 18

Business Continuity & Disaster Recovery (BCDR)

Resilience

BCDR is the set of processes and plans that help an organization return to normal business operations when a disaster strikes — natural, technical, or cyber. While closely related, BCP and DRP describe two subtly different approaches.

BCP vs DRP — The Core Distinction

AspectBusiness Continuity Plan (BCP)Disaster Recovery Plan (DRP)
ApproachProactiveReactive
FocusMaintain operations before, during, afterRestore IT systems and data after
ScopeWhole business — people, processes, partnersMainly IT infrastructure
Time horizonStrategic / long-termTactical / immediate
RelationshipDRP is typically a subset of BCP.

Two Critical Metrics — RTO & RPO

RTOdowntime
Recovery Time Objective — Maximum acceptable downtime before business is seriously hurt. Example: payment system RTO = 1 hour.
RPOdata loss
Recovery Point Objective — Maximum acceptable data loss measured in time. Backup every 15 minutes → RPO = 15 minutes.

The BCDR Planning Process

1Risk & Business Impact Analysis (BIA)
Identify potential threats — natural disasters, ransomware, system failure, pandemic, supply-chain disruption. Assess likelihood and impact on each business function.
2Identify Critical Assets & Processes
Which functions MUST run? Which can pause for hours, days? Map dependencies — equipment, staff, suppliers.
3Set RTO / RPO Targets
Define recovery objectives per critical asset. Drives technology choices (backup frequency, failover).
4Develop Strategies
Failover sites (hot/warm/cold), cloud backups, virtual machines, redundant networks, alternate communication channels.
5Assign Roles & Responsibilities
Executive sponsors, IT leads, department heads, BCDR team. Communication fallbacks (Wi-Fi/cellular may be down).
6Document Procedures
Step-by-step playbooks — easy to follow under pressure, accessible offline.
7Test & Refine
Tabletop exercises → walk-throughs → simulations → full-scale drills. Discover gaps in calm before chaos.

Types of Recovery Sites

  • Hot site — Fully operational mirror, ready to take over instantly. Most expensive, lowest RTO.
  • Warm site — Partially configured, needs data sync. Moderate cost & RTO.
  • Cold site — Empty space with utilities. Cheapest, highest RTO (days to set up).
  • Cloud / Virtualized recovery — VMs spin up on demand. Increasingly the default.

Types of DR Plans

Data Center DR
Restoration of physical or virtual data center after disaster.
Network DR
Restore internet, LAN/WAN, cellular connectivity.
Cloud DR
Replicate workloads to cloud regions; failover if primary fails.
Virtualized DR
VMs as recovery targets — minimal setup, high availability.

Testing Methods

  1. Tabletop exercise — Team walks through the plan verbally.
  2. Structured walk-through — Members role-play their responsibilities.
  3. Simulation — Replicate emergencies in a controlled environment.
  4. Parallel test — Run recovery systems alongside production.
  5. Full interruption — Actually shut down production, fail over. Most realistic, most disruptive.

A weakness threats can exploit (unpatched software, weak passwords).
Likelihood
Probability of the threat exploiting the vulnerability.
Impact
Consequence if it happens — financial, operational, reputational, legal.
Control
Countermeasure that reduces risk (firewall, MFA, training).
Residual risk
Risk that remains after controls are applied. Zero risk doesn't exist.

The Risk Management Lifecycle

1Identify
Catalog assets, threats, vulnerabilities. Know what you have and what could go wrong.
2Assess / Analyze
Score likelihood and impact for each risk. Qualitative (high/med/low) or quantitative (₹ amounts).
3Prioritize
Rank risks. Focus on those with high likelihood + high impact first.
4Treat / Mitigate
Choose a response strategy (see below) and implement controls.
5Monitor & Review
Continuous monitoring. Risks change as threats, business, and tech evolve.

The Four Risk Treatment Strategies

Mitigatereduce
Implement controls to reduce likelihood or impact. The default — patch systems, train users, deploy firewalls.
Transfershift
Shift the risk to a third party — cyber insurance, outsource to managed services, contractual indemnity.
Avoideliminate
Eliminate the activity causing the risk — don't process sensitive data you don't need, decommission risky systems.
Accepttolerate
Acknowledge and absorb the risk — when mitigation cost exceeds expected loss, or it's small enough to live with.

Data Privacy

Rights & Compliance

Data privacy is the right of individuals to control how their personal data is collected, used, stored, and shared. It's about ensuring people retain meaningful agency over information about themselves in a digital economy that runs on data.

Privacy ≠ Security — A Critical Distinction

AspectData SecurityData Privacy
FocusProtecting data from unauthorized accessProtecting individual rights over their data
QuestionIs this data safe?Should we even have this data?

Privacy requires security — but security alone doesn't ensure privacy. A perfectly secured database of data you shouldn't have collected is still a privacy violation.

The Core Privacy Principles

Most privacy laws are built on a common set of principles, codified by the OECD and adopted in GDPR, CCPA, and India's DPDP:

Lawfulness & Fairnessbasis
Process data only with a valid legal basis (consent, contract, legal obligation, vital interest, public task, legitimate interest).
Purpose Limitationspecific
Collect for clearly stated purposes; don't reuse for unrelated reasons later.
Data Minimizationminimal
Collect only what's necessary. No "just in case" hoarding.
Accuracyquality
Keep data accurate and up to date. Allow correction.
Storage Limitationretention
Keep data only as long as needed for the purpose; then delete.
Integrity & Confidentialitysecurity
Protect data via appropriate technical and organizational measures.
Accountabilityprove it
Demonstrate compliance with documentation, audits, DPIAs.
Transparencynotice
Tell data subjects clearly what you collect, why, who gets it, how long you keep it.

Categories of Personal Data

Personal Data (PD)
Any data identifying an individual — name, email, phone, IP address, device ID.
Sensitive Personal Data
Higher-risk categories — health, biometrics, financial, sexual orientation, religious/political beliefs, genetic data.
PII
Personally Identifiable Information — US term, roughly equivalent to PD.
Pseudonymized
Identifier replaced with alias; can be re-identified with the key. Still personal data.
Anonymized
Truly unrecoverable identification removal. No longer personal data — generally out of scope of privacy laws.

Common Data Subject Rights

  • Right to Access — Get a copy of your data + processing details.
  • Right to Rectification — Correct inaccurate data.
  • Right to Erasure ("Right to be Forgotten") — Have data deleted.
  • Right to Restriction — Pause processing temporarily.
  • Right to Portability — Get your data in machine-readable form for transfer.
  • Right to Object — Opt out of certain processing (e.g., marketing, profiling).
  • Right to Withdraw Consent — Withdraw a previously given consent at any time.
  • Right to Grievance Redressal — Lodge complaints with the regulator.

Privacy by Design — The Engineering Mindset

Coined by Ann Cavoukian; codified in GDPR. Seven foundational principles:

  1. Proactive not reactive; preventive not remedial.
  2. Privacy as the default setting.
  3. Privacy embedded into design.
  4. Full functionality — positive-sum, not zero-sum (you can have privacy AND functionality).
  5. End-to-end security — full lifecycle protection.
  6. Visibility and transparency.
  7. Respect for user privacy — keep it user-centric.

Privacy-Enhancing Technologies (PETs)

  • Encryption — Protect data at rest and in transit.
  • Tokenization & Masking — Replace sensitive data with non-sensitive equivalents.
  • Differential Privacy — Add noise to statistical outputs so individuals can't be re-identified.
  • Federated Learning — Train ML models without centralizing raw data.
  • Homomorphic Encryption — Compute on encrypted data without decrypting.
  • Zero-Knowledge Proofs — Prove a fact without revealing the underlying data.
key foundation

§ 21

Digital Personal Data Protection Act, 2023

India · DPDP

India's DPDP Act 2023 (also called DPDPA-2023) is the country's first standalone data protection law. It governs how digital personal data is processed, balancing individual rights with the legitimate need to process data. It replaces the patchwork that previously sat under the IT Act §43A and SPDI Rules, 2011.

historical note

The DPDP Act is the first Act of the Parliament of India to use "she/her" pronouns instead of the usual "he/him." A small but historically significant choice.

Scope & Applicability

  • Applies to digital personal data processed within India.
  • Covers data collected offline and later digitized.
  • Extraterritorial: also applies to processing outside India if it involves offering goods/services to data principals in India.
  • Does NOT apply to: non-digital data, non-personal data, processing for personal/domestic purposes, publicly available data published by the individual.
  • Unlike GDPR: DPDP does not distinguish between personal and sensitive personal data — all digital personal data treated under one regime.

Key Entities — The Vocabulary

Data Principalperson
The individual the data is about. GDPR equivalent: data subject. This is YOU.
Data Fiduciarycontroller
Anyone (person, company, gov entity) who alone or jointly determines the purpose & means of processing. GDPR equivalent: data controller.
Significant Data Fiduciary (SDF)major
Larger fiduciaries identified by Govt based on volume & sensitivity of data, risk of harm, sovereignty concerns. Heavier obligations.
Data Processorvendor
Processes data on behalf of a fiduciary (e.g., a cloud provider).
Consent Managerbroker
A registered platform allowing principals to give, manage, review & withdraw consent. Acts as accountable intermediary.

Rights of the Data Principal (You)

  1. Right to Information — Know what data is being processed, by whom, with whom it's shared.
  2. Right to Correction & Erasure — Get your data corrected, completed, updated, or erased when purpose is fulfilled.
  3. Right to Grievance Redressal — File complaints with the fiduciary first, then the Data Protection Board.
  4. Right to Nominate — Nominate someone to exercise your rights in case of your death or incapacity. (Unique to DPDP — not in GDPR.)
  5. Right to Withdraw Consent — Withdraw consent at any time.

Duties of the Data Principal — Yes, You Have Duties

Unlike most global privacy laws, DPDP imposes duties on individuals too:

  • Comply with all applicable laws when exercising rights.
  • Don't impersonate another person when providing data.
  • Don't suppress material information.
  • Don't register false or frivolous complaints.
  • Furnish only authentic information.

Violations attract a penalty of up to ₹10,000.

Obligations of Data Fiduciaries

  • Obtain free, specific, informed, unconditional consent before processing (unless processing is for a "legitimate use" defined in the Act).
  • Provide a clear, accessible privacy notice in English and 22 scheduled Indian languages.
  • Ensure accuracy and completeness of data.
  • Build reasonable security safeguards — encryption, masking, access controls, regular audits.
  • Notify the Data Protection Board AND affected persons in case of personal data breach.
  • Erase personal data when the purpose is fulfilled, consent withdrawn, or legal retention period over.
  • Provide effective grievance redressal mechanism.

Extra Obligations for Significant Data Fiduciaries (SDFs)

DPO Appointment
Appoint a Data Protection Officer (DPO) based in India, answerable to the Board of Directors.
Independent Audits
Conduct independent data audits periodically.
DPIA
Conduct Data Protection Impact Assessments for risky processing.
Periodic Reports
Share significant observations & gaps with the Data Protection Board.

Children's Data — Stringent Rules

  • A "child" = anyone under 18 years.
  • Must obtain verifiable parental/guardian consent before processing.
  • Cannot do behavioral tracking, monitoring, or targeted advertising aimed at children.
  • Processing must not be detrimental to the child's well-being.
  • Same protections extend to persons with disabilities.

Cross-Border Data Transfer

Transfer of personal data outside India is generally permitted, except to countries the Central Government specifically restricts via notification. This is a more permissive approach than GDPR's "adequacy decision" model.

The Data Protection Board of India (DPB)

Independent body established under Section 18, headed by a Chairperson and Members appointed by the Central Government.

  • Adjudicates disputes between fiduciaries and principals.
  • Imposes penalties on non-compliant fiduciaries.
  • Directs fiduciaries to take urgent measures after a data breach.
  • Operates as a digital office for quick, transparent resolution.
  • Does NOT have power to issue guidance/regulations (that stays with Govt/MeitY).
